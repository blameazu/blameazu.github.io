什麼是 Splay Tree
#

Splay Tree，也稱為伸展樹

是一個平衡二元搜尋樹

通常你不會用到它

但在 Link Cut Tree、Euler Tour Tree 中會使用到

所以算是一個複雜演算法的先備知識

什麼是二元樹
#

有一棵樹，它的結構保證每棵子樹最多只擁有 左子樹、右子樹

也就是說每個節點最多有三的度數，out : 2、in : 1

什麼是二元搜尋樹
#

在二元樹的狀況下又多了一個限制

假設現在有一棵子樹，父節點為 $A$

左子樹節點為 $B$

右子樹節點為 $C$

那麼存在兩個性質

  • $A$ > $B$，也就是說左子樹的所有節點必小於父子節點
  • $A$ < $C$，也就是說右子樹的所有節點必大於父子節點

這兩個性質全部存在於所有子樹，那麼我們就可以稱之為二元搜尋樹

btw 二元搜尋樹因為有上面的好酷的性質，所以很好搜尋

複雜度瓶頸
#

蛤假如我今天要搜尋一個節點 $x$ 好了

他有可能是一條超級長的鏈的葉子節點欸!

這樣我們搜尋最糟可以到 $O(\text{鍊長})$ 也就是 $O(N)$

超爛的欸，我們要怎麼辦才能讓他的複雜度變好呢?

沒錯，這就是平衡樹的精隨了

改善樹型結構
#

既然我們搜尋東西每次都會花費鍊長(深度)

那麼我們就感性的把所有詢問頻率次數高的節點往上搬

那這樣我們的深度就可以有感的減少

這邊先不要管合不合理，反正結論是好的哈哈

而改善樹型結構的方法有很多

像是建樹時用隨機值來均攤深度

對，上面說的就是 treap

但我們今天講的是 Splay Tree!

Splay Tree 反而沒有從建構的角度下去思考

他是採用旋轉子樹根結點的方式

在不影響二元搜尋樹的性質下，盡可能的讓詢問複雜度均攤下來是好的

旋轉
#

以下的方法應該不是正常人可以獨自推敲出來的 所以建議就直接背起來 or 感性理解一下

zig、zag
#

定義右旋為 zig，左旋為 zag

在保證二元搜尋樹的性質下將原本是 $y$ 為根的二元搜尋樹 中的 $x$ 轉為根

而二元搜尋樹的性質也可以想像成旋轉後的樹形結構的中序不變

蛤? 旋轉後整棵樹就長不一樣了吧? 為什麼還叫做旋轉?

痾 應該是這個就是旋轉，但是在旋轉的過程中保證性質下必須拆邊以及連邊，反正定義就長這樣owo

蛤? 我要怎麼在實作中快速的想出我要誰連誰，斷誰的邊?

我們可以想像一個中間過程

我們會先將 $y$ 往下放至與 $x$ 的子樹同高

接著因為已經違反二元樹的性質(現在是三個子樹)

所以就理所當然地把 $B$ 丟過去給 $y$ 連了

阿 zag 只是反過來而已，差不多

rotate
#

根據上述所說的 zig、zag 進行實作

我們可以發現 zig、zag 會依據父節點與子節點的關係進行操作

也就是說我們其實可以把 zig、zag 寫在一個 rotate 函式中就行

struct Node {
    Node* l = 0;
    Node* r = 0;
    Node* fa = 0;
    void rotate() {
        if(fa == 0) return; // 已經在 root 了
        
    }
};

這樣就好了嗎?
#

我們要轉 $x$ 節點至根結點時

我們只需要判斷他與父子節點的左右子樹關係

然後再做 zig or zag 就行了嗎?

答案是不行的，這樣複雜度不好…

你可以證證看複雜度
